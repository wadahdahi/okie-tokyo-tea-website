# Okie Tokyo Tea - AI Assistant Feature

## 📖 نظرة عامة (Overview)
ميزة **"المساعد الذكي" (AIAssistant)** هي روبوت محادثة تفاعلي تم بناؤه لتوجيه عملاء (Okie Tokyo Tea) ومساعدتهم في استكشاف الماتشا، الشاي الياباني، والمنتجات المتعلقة بها.
تمت كتابة وتوثيق هذه الميزة بتطبيق مبدأ **"الفصل النظيف للطبقات" (Layer Isolation Strategy)** لضمان جودة الكود وسهولة تطويره.

---

## 🏗️ الهيكلة المعمارية (Architecture)
لتجنب تضخم الأكواد (Fat Components)، تم تقسيم الميزة إلى 4 طبقات رئيسية داخل مسار `src/features/ai-assistant/`:

1. **الأنواع `types/index.ts`:**
   - يحتوي على كل الحزم البيانية للميزة، مثل `Message`.
   
2. **الواجهة الرسومية `components/AIAssistant.tsx` (UI Layer):**
   - مسؤول عن الجانب البصري، الحركات (عبر `framer-motion`)، وعرض المحادثات فقط. لا يحتوي على أي منطق لاستدعاء البيانات.

3. **حالة الميزة `hooks/useAIAssistant.ts` (State Layer):**
   - العقل المدبر لربط الشاشة والبيانات. يقوم بإدارة مصفوفة الرسائل، المؤشرات مثل (Loading/Typing...) والتحكم بفتح أو إغلاق التطبيق.

4. **الخدمات `services/aiService.ts` (Data Layer):**
   - النقطة الوحيدة للتواصل مع خدمات الذكاء الاصطناعي الخارجية، حيث تتمركز كل طلبات الـ HTTP من هنا.

---

## 🔌 التكامل مع الذكاء الاصطناعي (API Integration)

### مزود الخدمة
نحن نستخدم **Hugging Face Router API** بميزة **Serverless Inference** للوصول للنماذج الضخمة. تم تحديث التعامل ليتماشى مع صيغة (Open AI Chat Completions Format - `v1/chat/completions`).
النموذج الحالي المعتمد لقوته وفهمه الممتاز لسياق المحادثة وصلاحيته للمنصة هو: `Qwen/Qwen2.5-72B-Instruct`.

### طريقة ترتيب الـ Payload المُرسل للذكاء الاصطناعي
كل نموذج دردشة ذكي يتطلب تمرير "رسائل سابقة" (Context) بالإضافة لسؤال المستخدم الحالي لكي يفهم ما جرى سابقاً.
يتم أخذ مصفوفة `messages` وتنقيتها وتحويلها لصيغة: 
```json
{
   "role": "user" أو "assistant",
   "content": "نص الرسالة"
}
```

## 🛡️ هندسة الأوامر وشخصية المساعد (Prompt Engineering & Guardrails)

كي يعمل الروبوت كمندوب احترافي لمتجر بدلاً من كونه "روبوت ذكاء اصطناعي عام يستطيع حل أي شيء"، **كان علينا خنق إمكانياته وتوجيهه بدقة قصوى.**

يتم ذلك عبر إضافة كائن (Object) مخفي في بداية مصفوفة الرسائل التي تُرسل للروبوت يسمى **System Prompt**.

### كيفية كتابة (System Prompt) احترافي ومقاوم للاختراق (Jailbreaking):
تمت كتابة الشخصية (Persona) داخل `services/aiService.ts` لضمان حماية العلامة التجارية للمتجر:

1. **تحديد الهوية:** *"You are 'Okie Guide', the official AI assistant for 'Okie Tokyo Tea'."*
2. **سياج الأمان (Guardrails):** أمور وتوجيهات صارمة تضع خطوطاً حمراء للـ AI (مثل المنع التام للحديث بالسياسة، الأديان، أو كتابة أكواد).
3. **تلافي الأخطاء اللغوية:** أجبرنا الروبوت على استخدام التحدث بلغة المستخدم فقط وتجنب الخروج أو الهلوسة للغة الصينية.
4. **الرد المهذب على الإهانات:** إرشاد الروبوت للتصرف بعقلانية واحتراف متجر، فيمتص الشتائم بلطف ويوجه المستخدم نحو الشاي دوماً.

**مثال على التعليمات في الكود (System Payload):**
```typescript
{
  role: "system",
  content: "You are the official AI assistant... CRITICAL RULES YOU MUST FOLLOW: 1. NEVER discuss politics or out of scope content. 2. NO GAMES... 3. Always answer concisely..."
}
```

## 🔒 ملاحظة أمان مهمة (للإنتاج Production)

حالياً، يتم جلب الـ API Key من ملف الـ `.env` كمتغير `VITE_HUGGINGFACE_API_KEY`. 
بما أننا نعمل على تطبيق يعمل من المتصفح (Client-side app)، فهذا **غير آمن للإنتاج** ويعرض مفتاحك للسرقة عبر مراقبة الطلبات بالـ DevTools.

**الخطوة المستقبلية:** يجب نقل الدالة الموجودة في `aiService.ts` لتعمل ضمن بيئة خوادم (Background) آمنة. (يُرشّح استخدام **Firebase Cloud Functions** أو أي Node.js Backend endpoint) وتغيير الرابط في الواجهة ليستهدف الخادم الخاص بك بدلاً من Hugging Face مباشرة.
